# 5장. 응집도
응집도란 모듈 내부에 있는 데이터와 로직 사이의 관계가 얼마나 강한지 나타내는 지표. 

Data와 Logic이 분리 되어있으면 응집도가 떨어진다. 
Static 메서드를 오용하면 응집도가 낮아진다. 
객체의 인스턴스 변수를 사용하는 함수여야된다는건데 
함수에 static 붙여보고 컴파일 문제가 되지 않는다면 인스턴스 변수를 쓰지 않는다는 거니 응집도를 고려해봐야한다. 

로그출력, 포맷변환 정도는 static으로 설계해도 좋다. 
횡단 관심사는 static으로 만들어도 좋다 

private생성자 + 팩토리 메서드로 초기화 하면 의도대로 인스턴스 변수를 가지며 응집된 로직도 가질 수 있다. 
생성로직이 너무 많으면 팩토리 클래스로 분리할수도 있다. 

~~~kotlin
class ActorManager {
	void shift(Location location) {
		location.x += 10 ;
	}
}
~~~
데이타는 Location인데 이걸 매개변수로 받고 로직으로 처리하는 클래스가 있다. 
응집도가 낮다. 
그리고 외부에서 변경 됐다는걸 알 수 없다.

함수의 내용을 하나하나 확인하게 만드는 구조는 로직을 읽고 이해하는 데 시간이 오래 걸리고 가독성에 좋지 못하다. 

그래서 위 클래스는 
~~~kotlin
	class Location {
		void shift() {
			this.x += 10 ;
		}
	}
~~~
로 변경 할 수 있다. 

매개변수가 많다는 얘기는 처리할게 많은 함수라는 얘기
